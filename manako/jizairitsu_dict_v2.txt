■ 1. 自在律とは

【命名の意味】
  「自在」＝どんな展開にも対応できる融通無碍の計算機
  「律」＝感情を排した規律・法則

【設計哲学】
  - オッズを完全無視。レース結果のみに集中する
  - 感情バイアスを排除した純粋物理シミュレーター
  - 「買い目を出す機械」ではなく「展開の理を計算する観測機器」

【システム構成（読み込み順）】
  var App = {}          ← グローバルオブジェクト（インライン宣言必須）
  tamaki_speech.js      ← 七曜院パラドキサ環（たまき）UIコンポーネント
  kiyone_voice.js       ← キヨネ音声UI
  shakkou_donperi_display.js ← 赤口呑縁 表示層
  shakkou_donperi_core.js    ← 赤口呑縁 コア計算
  keirin_logic.js       ← 予想ロジック本体
  LogModule.js          ← GASへのログ送信
  meigen.js             ← 名言システム
  paste-input.js        ← レース情報自動取得UI

  ※ var App={} が欠けると全スクリプトが即死（地雷）

■ 2. 令種の概念

【令とは】
  自在律が出力する「推奨展開シナリオ」の種別。
  同一レースを複数の展開仮説で並行計算し、シナリオごとの評価を出す。

【令の種類と発動条件】

  ☀️ 晴天令（seitenrei）
    発動：常時
    意味：実力通りに決まりやすい安定展開のシミュレーション
    内容：3シナリオ（先行有利・捲り有利・差し有利）の統合スコア
    買い目：3連単・3連複
    荒天令バイアスは適用しない

  ⛈️ 荒天令（koutenrei）
    発動：常時（晴天令と並行計算）
    意味：風・疲労・プレッシャーで実力者が崩れる波乱展開のシミュレーション
    内容：晴天令の計算に「荒天令バイアス係数群（C_risk〜C_suicide）」を上乗せ
    買い目：3連複・2連単

  🌌 赤口呑縁（shakkou_donperi）
    発動：晴天令・荒天令の計算完了後に自動起動
    意味：1465通りの並行世界シミュレーション。「揺らぎ」の可視化
    内容：世界線ごとにカオスイベントを付加して試行し統計を取る
    出力：最頻出着順TOP3・生存濃度・因果分岐（イベント別最頻出目）

  ⚡ 壱耀晴乾ノ象（ichiyo）
    発動条件（全て同時成立が必要）：
      ① 天雲指数 = 33
      ② 風速 ≤ 2.0m
      ③ 晴天令1位の選手が「差マ（追）」かつ「◎印」
    意味：物理条件が整った穴差し選手への天命収束サイン
    出力：たまきが特定の選手番号を告げるセリフ

  ⚖️ 審眼八卦（shingan_hakke）
    発動：ユーザーが任意スイッチをONにした場合
    意味：指定した係数に×1.05の後乗せ補正を加えた独自ランキング表示
    対象係数：ライン強度（c_l）・競走得点（c_score_adj）・近況（c_recent）
             ・印（c_wmark）・展開タイプ（脚質絞り込み）・地元（自動）
    ※ 審眼八卦ON時はGASへのハズレ解析データ送信がスキップされる

■ 4. 全係数の意味と設計根拠

【コア計算式】
  final_score =
    score
    × c_score_adj   （得点補正）
    × c_wmark       （印補正）
    × c_recent      （近況補正）
    × c_s1          （スタンディング1位）
    × c_b1          （バック1位）
    × c_l           （ライン結束力）
    × c_e           （バンク脚質適性）
    × c_local       （地元補正）
    × physicalPenalty    ÷1 （直線短ペナルティ）
    ÷ cantoMakuriPenalty ÷1 （高カント捲りコスト）
    × warpBoost          （イン突きブースト）
    × kururu.adj         （風補正）

---

c_score_adj（得点補正）
  式 : 1.0 + (score÷100 - 1) × R_BIAS
  例 : score=110, S級(R_BIAS=1.15) → 1.0 + (1.10-1)×1.15 = 1.115
  意図: 得点が高い選手をより高く・低い選手をより低く評価するスケール調整

c_recent（近況補正）
  式 : (1.0 + (4 - avgRank)×0.05 + trendBonus) × RECENT_WEIGHT
       avgRank = 直近3走の着順平均
       trendBonus: 連続悪化（着順が悪化方向）=+0.03 / 連続改善=−0.03 / その他=0
       ※「悪化」とは着順数値が大きくなること（例: 1→2→3）
  意図: 平均1着≒+0.15、平均7着≒−0.15 の範囲で近況を反映

c_wmark（印補正）
  ◎ = 1.04  / 〇 = 1.02  / △ = 1.005
  ✕ = 1.003 / 無 = 1.00
  意図: 専門紙の格付けを微補正として乗算

c_s1（スタンディング1位補正）
  S1位 = 1.005 / それ以外 = 1.0
  意図: ライン先頭であることの象徴的優位

c_b1（バック1位補正）
  B1位 = 1.015 / それ以外 = 1.0
  意図: 先行を守りながら自身も着を狙う「二重負荷」への実戦的評価。B1位の方が加点大

c_l（ライン結束力係数）
  ■ 一般競輪（IS_GIRLS=false）
    先頭  : 1.0 + COOP_WEIGHT×0.05
            → S=1.06 / A=1.05 / チャレ=1.04
    2番手 : 1.0 + COOP_WEIGHT×0.04 × diffFactor
            diffFactor = 先頭との得点差に応じて変動
              差10以上=0.3 / 差5以上=0.6 / 差0以上=1.0 / 番手が上回る場合=1.3
    3番手以降 : 1.00（補正なし）
    競り敗者  : c_l適用スキップ

  ■ ガールズ競輪（IS_GIRLS=true）
    2番手 : エースマーク連動
              ◎ = 1.12 / 〇 = 1.08 / それ以外 = 1.03
    3番手以降: 2番手値の半分補正
    競り敗者 : スキップ

c_e（バンク脚質適性）
  bankdata.jsonの keirin_bias から取得
    脚質→バイアスキー対応:
      逃（逃先）→「先行」 / 捲り→「捲り」 / 差マ（追）→「差し」
  例 : 大宮バンク / 差マ → keirin_bias["差し"] = 1.10

c_local（地元補正）
  地元チェック = 1.03 / それ以外 = 1.0
  ※ 審眼八卦で地元選手がいる場合は自動的にSNGN補正も追加される

physicalPenalty（直線短ペナルティ）
  対象: 4番手以降の選手
  判定基準: bankdata.straight_deviation（みなし直線）
    < 35m : 0.80 〜 最低0.70（深さに応じ）
    35〜50m: 0.93 〜 最低0.87
    ≥ 50m : 1.00（ペナルティなし）

cantoMakuriPenalty（高カントペナルティ）
  対象: 捲り系選手（style='自'）
  発動: bankdata.canto > 32度
  値  : 1.25（÷として作用するため実質的に×0.80）
  意図: カントが深いバンクは捲りのエネルギー消費が大きい

warpBoost（イン突きブースト）
  発動: 競りでイン(番手)が勝利 かつ その選手のラインポジションが2番手
  対象: 全体3〜4番手の選手
  値  : 1.35
  意図: 番手が競りを制してインを突いた際の経験則的優位（約35%）

■ 5. 風補正（kururu / getKururuAdjustment）

  実効風速 v = windSpeed × beta（bankdata.betaで調整）

  風圧係数kpの計算:
    v ≤ 3.0m : kp = v × 0.05
    v ≤ 7.0m : kp = 0.15 + (v-3)^1.8 × 0.085
    v > 7.0m : kp = 0.51 + (v-7)^3.0 × 0.3

  ライン位置シールド（positionShield）:
    先行（先頭） = 1.00（遮蔽なし・直接風を受ける）
    番手         = 0.65（後方に守られる）
    3番手以降    = 0.50（さらに守られる）

  最終補正係数:
    finalAdj = 1.0 + (wind_vector × kp × alpha × positionShield)
    wind_vector: 風向きがH向かい風=-1.0 / H追い風=+1.0 / 横風=±0.2

  無効化条件: 風速≤1.0m / 風向き='無風'/'none' / 屋内バンク(alpha=0)

■ 6. 競り補正（applySeriCorrection）

  競り係数（seri_coef）で勝者を予測:
    seri_coef = score × SERI_STYLE_BONUS × (◎なら×1.04)
    SERI_STYLE_BONUS: 逃=1.08 / 追=1.05 / 自=0.95

  競り勝者 : final_score × (1+0.05) × (1-0.15) ＝ 実質×0.8925
  競り敗者 : final_score × (1-0.25) ＝ ×0.75
  ※ 競り敗者は c_l も適用スキップ（計算上ほぼ圏外扱い）

■ 7. 荒天令バイアス係数群（calculate_koutenrei_bias）

  荒天令シミュレーション専用。晴天令には一切適用しない。
  以下の係数を順番に乗算する（複数同時発動あり）。

  C_risk
    条件: score < 平均得点-2 かつ 直近3走平均着順 ≥ 4.0
    効果: × 0.97
    意味: 地力も近況も劣る選手の波乱時さらなる沈み

  C_mental
    条件: S級 or 最高得点選手 かつ 直近1着スタート（recent先頭='1'）
    効果: × (1.0 - v×0.005)   ← 風速が強いほど下がる
    意味: 狙われる本命がプレッシャーと風で崩れやすくなる

  C_recovery
    条件: style=追 かつ scoreDiffRatio（得点の上位率）> 0.6
    効果: × (1.04 + (scoreDiffRatio-0.6)×0.1)
    意味: 実力のある差し選手は乱戦で伸びしろがある

  C_target
    条件: 最高得点選手 かつ 自走型ライバル（逃/自）≥ 2本
    効果: × (1.0 - v×0.007)   ← 風速が強いほど下がる
    意味: 強い逃げ選手は複数ラインに追われると崩れやすい

  C_split
    条件: ライン内 先頭と2番手の得点差比 ≥ 0.30
    効果: × (1.0 - (差比-0.30)×0.15)  ← 2番手に適用
    意味: 先頭と番手に実力差があると番手が連れ込めない

  C_pace
    条件: 逃げ/捲り系の最高得点選手 得点≥105 かつ 競合ライン≥2
    効果: × 0.96  ← 先頭に適用
    意味: 超高得点の逃げ選手も複数ラインに追いつかれやすい

  C_timing
    条件: style=自（捲り）かつ ライン内番手ポジション（先頭でない）
    効果: × 0.97
    意味: 捲り選手が番手に入ると仕掛けどころを失う

  C_guard
    条件: 番手選手が低得点（下位40%以下）+ 先行型が多い
    効果: × 0.95（低得点）× 0.95（先行型≥2）← 最大×0.9025
    意味: 番手が低得点で守られる状況が少ないと連れ込み怪しい

  C_suicide（自滅ライン）
    条件: 3人以上ライン + 全員に印（◎〇△）+ 自走型選手在籍
    効果: 自滅ライン員 → × SUICIDE_LIMIT（風速に応じて動的）
          非自滅ライン員 → × 1.05（漁夫の利）
          単騎選手 → × 1.02
    SUICIDE_LIMIT: S=0.97 / A=0.93 / チャレ=0.90 / ガールズ=1.00

  C_cap（下限キャップ）
    条件: final_score ÷ score < 0.85
    効果: final_score = score × 0.85 に強制修正
    意味: どんな波乱係数の積み重ねでも85%を割らせない

■ 8. 展開モード自動判定（runScenarioSimulation内）

  以下の変数を用いてレースの展開性質を自動判定:
    escapeMax  = 逃げ系選手の最高得点
    chaseMax   = 差マ系選手の最高得点
    makuriMax  = 捲り系選手の最高得点
    scoreGap   = chaseMax - escapeMax
    makuriVsNige = makuriMax - escapeMax

  判定結果 → 展開モード → 後乗せ係数（tenkaiBonus）:
    makuriVsNige ≥ 5  → 「捲」モード → 捲り系×1.15
    scoreGap≤0 かつ makuriVsNige≤0 → 「逃」モード → 逃×1.15・捲×1.10
    それ以外           → 「差」モード → 追加補正なし

  ※ 逃げ系0人の場合、並び先頭の選手を逃げ扱いとして展開判定に使用

【3シナリオのcD係数】
  先行有利 : 自×1.05 / 追×1.02
  捲り有利 : 自×1.00 / 追×1.05
  差し有利 : 自×0.90 / 追×1.08

【ラインカウントボーナス（applyLineCountBonus）】
  4人ライン : ×1.08
  3人ライン : ×1.04
  それ以外  : ×1.00
  ※ 晴天令スコアにのみ適用。荒天令には未適用

■ 9. 天雲指数（calculateTenunIndex）

  算出ロジック:
    晴天令TOP3のIDセット vs 荒天令TOP3のIDセット の一致数をカウント
      3一致 = 指数0   （「穏やかな空」完全安定）
      2一致 = 指数33  （「少し雲」やや混沌）
      1一致 = 指数67  （「荒れ気配」波乱含み）
      0一致 = 指数100 （「不測」完全混沌）

  たまきの反応（TAMAKI_SPEECHES）:
    index_0   : 穏やかなセリフ（表情: smile）
    index_33  : やや不安のセリフ（normal）
    index_67  : 波乱含みのセリフ（worried）
    index_100 : 「不測」「異常値」系セリフ（shocked）
    ichiyo    : 壱耀晴乾ノ象発動時の特別セリフ（smile+選手番号埋め込み）

  壱耀晴乾ノ象 発動条件（calculateTenunIndex内）:
    ① 天雲指数 = 33
    ② 風速 ≤ 2.0m
    ③ 晴天令1位の選手が差マ（style='追'）かつ◎印（wmark='◎'）

■ 10. bankdata.json 全パラメータ

  length             : バンク全長（m）。400m/500mの2種
  straight           : 名目上の直線長（m）
  straight_deviation : 物理ペナルティ計算に使う実測直線長（これが主役）
  canto              : カント角（度）。32度超で捲りコスト増大
  width_home         : ホーム幅（m）
  width_back         : バック幅（m）
  width_center       : センター幅（m）
  alpha              : 風の影響スケール係数（屋内バンクは0.0）
  indoor             : 屋内フラグ（true=wind_direction_map=null）
  beta               : 実効風速スケール（v = speed × beta）
                       バンクの地形・遮蔽物による風の弱まりを表現
  keirin_bias        : 脚質別バンク適性係数
                       {"先行": x, "捲り": x, "差し": x}
                       1.0を中心に±0.05〜0.10程度で設定
  wind_direction_map : 風向き → 影響タイプの対応表
                       値の種類: "H向かい風" / "H追い風" / "H→B横風" / "B→H横風"

■ 11. 買い目の導出ロジック

【晴天令買い目（generateSeitenreiBets）】
  ランキング順に r[0]=1位, r[1]=2位, r[2]=3位 を選出。
  r[2]の選出優先順:
    1) 差マ(追) かつ (△か◎) → 採用
    2) 差マ(追) のみ → 採用
    3) スコア3位 → 採用
    ※ r[0]・r[1] と重複する場合は次候補へ

  出力:
    3連単: r[0]-r[1]-r[2] / r[0]-r[2]-r[1] / r[1]-r[0]-r[2]
    3連複: 上位3人の組み合わせ＋上位4人目を含む組み合わせ

【荒天令買い目（generateKoutenreiBets）】
  ベース: 晴天令の r[0]/r[1]/r[2] をABC軸として使用
  特異点L（targetL）の選出（V10.4実装）:
    優先順1: 荒天令上位 かつ 各ラインTOP選手
    優先順2: 荒天令上位の先頭選手
    優先順3: フォールバック（荒天令2位）
  出力:
    3連複: A=B=C の組み合わせ
    2連単: A→B / A→C / B→A 他

【赤口呑縁の買い目活用（ユーザー向け）】
  出現出目TOP1: 最も収束しやすいパターン → 本命固めに使う
  生存濃度: 2着・3着入線が多い選手 → ヒモ選択に使う
  因果分岐: 特定カオスイベント発生時の出目 → 波乱保険として使う
  印の基準（formatter.js / strengthMark）:
    ◎ = 勝率≥28% かつ 3着内率≥70%
    ○ = 勝率≥20% かつ 3着内率≥60%
    ▲ = 勝率≥15% かつ 3着内率≥50%
    △ = 勝率≥10% かつ 3着内率≥40%
    × = 勝率≥5%
    ・ = 勝率5%未満

■ 12. 赤口呑縁（invokeShakkouDonperi）詳細

【1465の意味】
  Marvelの『インフィニティ・ウォー』でドクター・ストレンジが観測した
  並行世界数（1400万605通り）へのオマージュ。
  「競輪における唯一の正解を探す儀術」として設定。

【処理の流れ】
  1. 1465回をBATCH_SIZE=50件ずつ非同期分割処理
  2. 各試行で世界線を1本抽選（selectWorldLine）
  3. basePlayers をディープコピーして各係数を再計算
  4. 物理層 → 展開層 → kururu → 競り補正 → 荒天令バイアス を適用
  5. カオスイベントを付加（applyChaos）
  6. ゆらぎを乗算（flutter: grade別にmin〜min+range の乱数）
     s-kyu: 0.90〜1.10 / a-kyu: 0.85〜1.15
     girls: 0.85〜1.15 / a-chal: 0.80〜1.20
  7. スコア順にソートして着順を確定
  8. 全試行の結果をcosmosObserverで集計

【世界線テーブル（SHAKKOU_WORLD_TABLE）】
  ガールズ: 7種  W0〜W6（物理純正 733/1465≒50%が支配）
  S級     : 8種  W0〜W7（物理純正 879/1465≒60%が支配）
  A級     : 9種  W0〜W8（物理純正 586/1465≒40%）
  A級チャレ: 11種 W0〜W10（物理純正 249/1465≒17%、最混沌）

【カオスイベント一覧（applyChaos）】
  line_breakdown_mild : ライン微崩壊（先頭×0.85、番手が±10%ランダム、外差し×1.15）
  line_breakdown      : ライン崩壊（先頭×0.65、番手60%で×1.25 or ×0.75、外差し×1.30）
  line_breakdown_severe: ライン完全崩壊（全員×0.50、外差し単騎×1.50）
  early_attack        : 早仕掛け（成功率60%、成功→×1.40 失敗→×0.60）
  early_attack_success: 早仕掛け成功（確定成功、×1.40）
  early_attack_fail   : 早仕掛け失敗（確定失敗、×0.60）
  reckless_attack     : 無謀な早仕掛け（捲り系が×0.40、外差し×1.30）
  wind_gust           : 突風（1/3確率で追い風→先行系有利、1/3で向かい風→差し有利）
  seri_upset          : 競り番狂わせ（ランダム1選手×1.25）
  crash               : 落車（ランダム1選手×0.20）
  dark_horse          : ダークホース覚醒（得点85未満からランダム×1.50）
  rookie_rampage      : ルーキー暴走（得点90未満からランダム、40%で×1.60 / 60%で×0.50）
  makuri_suicide      : 捲り自滅（捲り系×0.45、外差し×1.20）
  escape_success      : 逃げ切り成功（逃げ系1人が×1.50、他全員×0.85）

【cosmosObserver（結果集計）】
  出力データ:
    statistics: 選手ごとの勝率・3着内率・平均着順・標準偏差
    topPatterns: 最頻出3着順 TOP3（出現回数付き）
    eventStats: イベント別発生回数・最頻出目
    worldStats: 世界線別出現回数

■ 13. GCボタン（ゴールドキャップ補正）

  手動GC（UIボタン）:
    score < 95.0 の場合に強制的に score = 95.0 へ引き上げ
    意図: A級平均スコア≒95を基準とした地力再定義

  自動GC（keirin-proxy-ii / A.L.L.のみ）:
    条件: 129期以降 かつ 25歳以下 かつ score < 80
          かつ recent中1着が3回以上
    効果: score = max(score, 90)
    意図: データ不足の若手選手の過小評価補正

  【注意】 score=0のルーキーにGC適用すると:
    c_score_adj=0.95（得点がないため下方バイアス）
    + scoreMax奪取でC_target誤発動
    + mainLine誤選出のリスクあり
    → 使用は慎重に判断

■ 14. 自動版（keirin-proxy-ii / A.L.L.）処理フロー

  A.L.L. = Automata Lex Libera（自動投稿エージェント）

  STEP 1 【selector.js】 レース選出
    - 当日開催全場の全レースIDを取得
    - Phase1: ライトスクレイプ（betTimeのみ）
      betTime が現在時刻+15〜30分のものを候補に絞り込み
    - Phase2: 候補のみフルスクレイプ（winticket含む）
    - 再確認: 7車立てのみ通過
    - 最大2レースをランダム選出

  STEP 2 【scraper.js】 データ取得
    - Kdreams: 出走表（車番・選手名・脚質・得点・決まり手・直近成績）
    - winticket: 並び予想・S/B/wmark・レース名
    - 直近3走: table[2]を今開催→前場所→前々場所の順で遡り取得
      ※ 落車・棄権・失格 → "9" に変換
    - 脚質判定: 決まり手カウント（逃・捲・差・マ）の最多値を採用
      同数の場合は脚質未確定（ログに警告、手動補完が必要）
    - S/B1位: winticketのS・Bスタンダード値の最大選手
      同数の場合は並び予想の先頭順で前を優先

  STEP 3 【weather.js】 風情報取得
    - 競輪場の緯度経度 → jiz41-weather-proxy経由でopen-meteoのAPIを叩く
    - wind_speed_10m（km/h）→ m/s（0.5刻み四捨五入）に変換
    - 風速 < 1.0m → 「無風」として扱う

  STEP 4 【orchestrator.js】 GC自動判定 + 予想実行
    - 新人レース名判定: "新人/ルーキー/新鋭"等が含まれる → ROOKIE_SKIPエラー
    - 低スコア判定: score≤67点の選手が2名以上 → LOW_SCORE_SKIPエラー
    - GC自動補正（前述）
    - c_l計算（DOM非依存のため orchestrator内で独自ロジックを実行）
    - 各係数（c_score_adj / c_recent / c_wmark / c_s1 / c_b1 / c_e）付与
    - 晴天令・荒天令シミュレーション実行
    - ラインカウントボーナス適用（晴天令のみ）
    - 天雲指数計算
    - 買い目生成（seiten/kouten両方）
    - 赤口呑縁実行

  STEP 5 【formatter.js】 Discordフォーマット生成
    - Embedsフォーマット（title/color/fields/footer/timestamp）
    - 天雲指数→色コード対応: 0=緑/33=黄/67=橙/100=赤
    - GC発動選手をフィールドに追記
    - 壱耀晴乾ノ象発動時は専用フィールドを追加
    - 赤口呑縁の統計をフォーマット（生存濃度＋最頻出TOP3）

  STEP 6 【poster.js】 Supabaseへ記録
    - discord_postsテーブルにINSERT
    - フィールド: race_id / title / color / timestamp / fields(JSON)
                 has_gold_cap / has_ichiyo / has_shakkou（フラグ）
